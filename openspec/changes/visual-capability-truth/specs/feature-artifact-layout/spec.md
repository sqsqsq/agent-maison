# feature-artifact-layout Spec Delta

## ADDED Requirements

### Requirement: Vision trust artifacts are runner-owned and append-only

The feature tree SHALL gain: `<feature>/vision/capability-receipt.json` (runner-owned canary/binding receipts with provider/model/probe context), `<feature>/vision/artifact-attestations.jsonl` and `<feature>/vision/policy-downgrades.jsonl` (append-only, row-hash-chained vision ledgers — attestation records incl. supersede events), and `<feature>/goal-runs/<runId>/intermediate-rounds.journal.jsonl` (goal-mode intermediate visual rounds, hash-chained; stored per goal run — attempt ordinals repeat across runs, so a feature-shared journal would let a new run adopt or be poisoned by a previous run's same-numbered rows). Agents SHALL NOT create or mutate these runner-owned files; harness checks consume them read-only. Runner ownership of the two vision receipts SHALL be enforced by sequencing, not naming: after every spec agent invocation the runner deletes both receipt files, reissues them per grading/audit result, and appends a goal-run event carrying the receipt file's sha256; consumers SHALL trust a receipt only when the **last** event for that invoke is an issuing event whose sha256 matches the on-disk file (agent-forged files or event lines written during the invocation are superseded by the runner's post-invocation cleanup and event append; a receipt with no runner event anchor SHALL be rejected fail-closed). Runner ownership of the two vision ledgers SHALL be enforced by single-writer sequencing plus bracketing: in goal mode the agent-side harness SHALL compute counterevidence conclusions without writing the ledgers (only the runner-spawned gate harness — `MAISON_GOAL_GATE_HARNESS=1` — or non-goal interactive mode commits; a forged flag is still caught by the bracket); the runner snapshots both ledgers' sha256 before each agent invoke (appending a `vision_ledger_anchor` event) and re-checks after — any change inside the agent window SHALL append `vision_ledger_tamper` and HALT the phase; after each gate harness run the runner SHALL append a `post_harness` anchor and update the checkpoint so the last anchor always reflects the last legitimate write before any halt/early-exit. The checkpoint lives outside the agent's workspace write scope (user home dir; the `MAISON_GOAL_CHECKPOINT_DIR` override is stripped from agent env), is namespaced by project-identity hash + feature + runId (runIds carry a random suffix — second-precision timestamps collide across projects), binds the frozen manifest identity fields **and the canonical authorization subset** (`auth_subset_sha256` over stable-stringified `pre_authorized_mutations` — mandatory under a deployed key; `pre_authorized_mutations` is itself one of the manifest identity fields, so a downtime change to it surfaces as an unauthorized identity drift and halts unless explicitly covered by `--override-manifest`, i.e. runtime escalation never masquerades as pre-run authorization), is written by atomic replace, and carries writer authenticity via HMAC under the `MAISON_HMAC_*` model (`MAISON_HMAC_GOAL_CHECKPOINT` — never present in agent env, **and stripped from every child process that can execute host-authored code**: hvigor/ohpm build chains and device tool spawns run under `stripTrustAnchorEnv`, since hvigorfile.ts/build plugins are agent-producible and env inheritance would hand the key to the agent indirectly): when a deployment key is configured, a missing/failed MAC or corrupted/identity-mismatched checkpoint SHALL verify as `invalid` and fail closed with no bypass; without a key the state is `ok_unauthenticated` (explicit weak-trust event, never silent) **and a UI-relevant run SHALL NOT produce a clean-completion terminal state — `CHAIN_SLICE_COMPLETED` is capped to `AWAITING_HUMAN_REVIEW`** (writer authenticity is a completion precondition, not a log line). Checkpoint write failure SHALL abort fail-closed (no silent weak-trust degradation). Because the ledgers are feature-scoped while checkpoints are per-run, a **feature-level authenticated head** (`vision-heads/<projectHash>/<feature>.json` with monotonic generation, both ledger hashes, last_run_id, MAC) SHALL be verified at every fresh run and resume before any baseline anchoring — cross-run tampering between goal runs is a `mismatch` halt, and the runner SHALL NOT re-sign a tampered baseline; the head is updated (generation+1) at every legitimate write point, and the per-run checkpoint references the head generation. The verified head generation, migration history, **and the byte digest of the last write** live in **runner memory** after startup verification: write points take them from memory only (never re-read from disk — a deleted/forged on-disk anchor must not be re-signed) and before every overwrite verify the on-disk anchors' MAC/identity/generation **and exact byte digest** against the trusted memory state, halting on absence, drift, or replay (a legitimately-MAC'd older file of the same run whose bytes differ from the last in-memory digest is rejected — identity+MAC alone cannot catch same-run replay). Because in-memory digests do not survive a restart, a persistent **MAC'd append-only high-water-mark journal** (`vision-heads/<projectHash>/<feature>.hwm.jsonl`, one chained row per legitimate write recording generation + head digest + prev-row hash, monotonic generation) raises the bar for restart rollback: at startup the current head's generation/digest MUST equal the journal's high-water — the comparison is **strict and bidirectional**: head behind the high-water (or same generation with a differing digest) is the rollback/corruption halt, and head **ahead** of the high-water SHALL equally fail closed as `incomplete_anchor_commit` (the residue of a crash between the checkpoint write and the HWM append) — it SHALL NOT be silently accepted, because the next legitimate write would append a cross-generation row and launder the incomplete commit into normal history. The feature head SHALL declare HWM establishment (`hwm_declared`, MAC-protected, head schema 1.1): at startup a verified head that declares the HWM with the HWM file **absent** SHALL fail closed (deleting the whole HWM file must not launder into a fresh legitimate chain — absent is NOT silently reinitialized); a legacy 1.0 head without the declaration plus an absent HWM is a one-time explicit bootstrap (event-logged, and the next legitimate write promotes the head to the declaring schema); only head-and-HWM both absent is true first initialization. **Honest capability boundary (not cryptographic cross-restart anti-rollback):** the HWM journal sits in the same trust directory and permission boundary as head/checkpoint, so an attacker able to replay those three files can also tail-truncate the HWM to the matching old generation. The HWM therefore detects **uncoordinated rollback / accidental corruption / non-tail deletion/insertion/reorder/modification / chain breaks** (a restart that replays head+checkpoint but does not also truncate the HWM tail is caught) — it does NOT provide cryptographic freshness against a coordinated replay. True hardened anti-rollback requires an independent non-rewindable anchor (permission-isolated broker / remote append-only store / trusted monotonic counter) and is tracked as pending (task 3.9j); the shipped behavior and all messages state this honestly rather than claiming cross-restart anti-rollback. Key rotation's HWM quarantine SHALL be a transaction (prepared → quarantined → committed | rolled_back) with a **MAC'd reseal journal (v2)** that records the planned backup names, the old head/checkpoint/HWM byte hashes, and the authorizing receipt's object_hash **before the rename executes** (a crash between rename and the quarantined journal write must still leave the backups locatable), and the quarantine SHALL take sha-verified **copies of all three anchors** (head and checkpoint are copied before any new-key rewrite; the HWM is renamed) — the anchor-commit order is head → checkpoint → HWM, so a crash after head/checkpoint were rewritten under the new key but before the HWM first write would otherwise leave a **mixed three-anchor state** where restoring only the HWM makes the original receipt (bound to the old head/checkpoint bytes) permanently unverifiable; the old-HWM rename SHALL fail closed on error (never warn-and-continue), the new chain SHALL be re-verified immediately after its first write before the journal commits, and startup SHALL run a **content-based recovery** before any old-anchor hashing or receipt validation: completion is recognized only when the canonical HWM differs from the bound old chain **and the full three-anchor state passes the same gates as a normal startup** — the head verifies against the current ledger snapshot, the run's checkpoint verifies (present, MAC-valid, files matching, `head_generation` interlocked with the head), and the HWM high-water is strictly equal to the head (generation and digest) — only then is the journal committed retroactively; **any partial state SHALL NOT be committed** (a premature commit turns the journal terminal and permanently forfeits the rollback entitlement, degrading a backup-recoverable scene into manual disposition) — it falls through to the three-anchor rollback instead; every other non-terminal state rolls back by restoring **each** anchor whose current sha differs from its journal-bound old sha from its own backup (backup sha verified against the binding **before** restoring — a tampered backup must not launder itself through recovery; an anchor whose old sha is `absent` is restored by deleting the newly-written file), and `rolled_back` SHALL be recorded only after **all three anchors re-verify equal to the bound old hashes** — so the original reseal receipt is reusable across every crash window; any unrestorable anchor → fail closed (manual disposition). Re-entering the quarantine while a non-terminal journal exists SHALL throw (recovery must run first — overwriting a live transaction journal destroys the recovery baseline), and the crash-recovery path SHALL NOT depend on same-run identity or on re-presenting a receipt. Trust-anchor env stripping SHALL be **case-insensitive** (Windows env is case-insensitive; a lowercase `maison_hmac_*` is the same variable and must be removed) and SHALL cover HDC/device-tool spawns (the HDC executable is host-selectable via `HARNESS_HDC_EXE`/`HDC_EXE`). The per-run checkpoint additionally binds and re-verifies the frozen manifest **identity hash** and the current feature-head generation on resume. The manifest identity hash covers the run-invariant security fields (start/end phase, requirement, chain, fidelity, budget, dependency, unattended, pre_authorized_mutations) and excludes runner-volatile fields (adapter/provenance/created_at) — the whole-file hash cannot serve drift detection because `writeGoalManifest` legitimately rewrites the file each run. It is stored as a **per-field** hash map, and the drift check runs **inside the feature/run lock and before any side effect** (before local-writeback/`writeGoalManifest`/canary/preflight — a tampered manifest must neither pollute another run's event log via a pre-lock write nor execute first and halt after). The trusted frozen baseline is the **authenticated checkpoint's** stored `manifest_identity_fields` (MAC-protected SSOT, checkpoint schema 1.2 where the field map is mandatory — a 1.2 checkpoint without it is invalid); the `manifest_identity_rebase`/`run_start` events are audit-only projections and SHALL NOT be used as the trusted rebase baseline. A **legacy 1.1 checkpoint without per-field identity SHALL NOT silently become the new baseline**: if its aggregate manifest hash equals the current identity hash the runner performs a one-time schema migration (event-logged; the commit writes the 1.2 field map as the new SSOT); if it differs, the resume SHALL fail closed unless the operator explicitly passes `--override-manifest` (no per-field diff is possible against an aggregate, so only whole-manifest confirmation is meaningful) — an upgrade downtime window must not launder requirement/budget/allowed-tools edits. An **unauthenticated** (`ok_unauthenticated`, no deployment key) checkpoint baseline still participates in the drift comparison (it catches naive tampering that edits only the manifest), but it SHALL be treated as weak trust, never as a trusted SSOT: the resume SHALL additionally require an explicit ack (weak flag — terminal state capped — or a strong ack receipt), and the `pre_run_manifest` mutation-authorization source SHALL be downgraded to non-machine-trustable for the run (see the goal-runner delta) — terminal-state capping alone is not sufficient because the run would otherwise still execute under possibly-widened budget/unattended/pre-authorized-mutation fields. On drift, authorization is **field-level**: each changed field must be covered by a corresponding override (`--override-manifest` authorizes all fields; `--override-start`/`--override-end` authorize only `start_phase`/`end_phase`; an applied `--fidelity`/`--fidelity-receipt` authorizes only `fidelity`/`fidelity_receipt`, with tier up/down rules still enforced by the fidelity preflight) — a bare `--override-start` SHALL NOT wave through a `requirement`/`budget`/`pre_authorized_mutations` change. An authorized transition is a rebase: this run's commit writes the new identity fields into the checkpoint, which becomes the new SSOT so a later plain resume does not re-report drift; the `expectedManifestHash`/`currentAuthSubsetSha256` force-equal checks are removed from checkpoint verification (they would make an authorized rebase self-invalidate), the checkpoint/reseal bindings use the effective (rebase-aware) identity hash, and unauthorized drift halts fail-closed. Key upgrade/rotation has an explicit **reseal protocol**: an `invalid` head/checkpoint/HWM under a newly-configured or rotated key SHALL be re-sealed only via a trusted confirmation receipt (`action: vision_trust_reseal`, object_hash bound to project hash + feature + runId + both ledger hashes + the old head file's byte sha256 + the old checkpoint file's byte sha256 + **the old HWM file's byte sha256** + the current authorization-subset hash + **the effective (rebase-aware) manifest identity hash** — otherwise a human-signed reseal would let the model widen the manifest authorization subset, and after an identity rebase the receipt must bind the new requirement/budget/allowed-tools/fidelity); the weak flag SHALL NOT participate. On a successful reseal the runner SHALL transactionally quarantine the old HWM journal (rename to `*.rekey-<ts>.bak`) so the new key's chain starts fresh — otherwise appending a new-key row to an old-key chain would fail MAC verification on the first old-key row and dead-lock the key rotation. All trust-anchor materials (`MAISON_HMAC_*`, `MAISON_TRUST_REGISTRY`, `MAISON_GOAL_CHECKPOINT_DIR`) SHALL be stripped from **every** child process that can execute host-authored code — including the entire Python preparation chain (interpreter probe, `import hylyre`, venv creation, pip install whose cwd is the host project where a host `sitecustomize.py`/same-named module could read them), not only the final Hylyre invocation. **Resume order is verify-then-migrate**: head and checkpoint SHALL be verified before any migration decision (a chained ledger swapped for a plausible chainless row is a `mismatch`, not a migration trigger). Acknowledgement has two tiers: a trusted confirmation receipt (`action: vision_ledger_ack`, object_hash bound to project hash + feature + runId + both ledger hashes, registry-verified) is a strong ack; the bare `--ack-unverified-ledgers` flag is a weak ack — a model can synthesize a CLI flag, so weak ack requires a passing events-anchor comparison (with no anchor available it SHALL NOT proceed) and caps the terminal state to `AWAITING_HUMAN_REVIEW`. Legacy migration of pure chainless ledgers SHALL be transactional: build and fsync the full new chain in a temp file, verify chain integrity and import counts, then atomically quarantine the original (recording its sha256) and rename the temp into place — a crash between the renames is recovered on next startup, and any build/verify failure leaves the original untouched (manual disposition); the migration record (old sha256 → new sha256) SHALL be persisted into the protected checkpoint and **inherited across subsequent checkpoint writes** (later pre-invoke/post-harness updates merge, never overwrite, the migration history). Conservative semantics stand: only downgrade/contradicted rows are re-imported; legacy verified/supersede SHALL NOT be auto-promoted; mixed or unparsable files SHALL NOT be auto-repaired.

Enforcement: `harness/scripts/goal-runner.ts`, `harness/scripts/utils/critic-receipt-producer.ts`（verifyVlSigningChain）, `harness/config.ts`（路径解析）

#### Scenario: agent-forged receipts without a runner event anchor are rejected

- **WHEN** an agent process writes a capability receipt and a refs receipt for the current run/attempt, but no runner issuing event with a matching file sha256 exists as the last event for that invoke
- **THEN** the vl signing chain verification SHALL reject both receipts and `vl_multimodal` final signing SHALL fail closed

### Requirement: Contracts and test plans carry the new machine blocks

contracts.yaml SHALL accept `integration_points[]` (consumer_module/provider_module/requires_modification/entry_symbol) normalized at load boundary like modules[]; test-plan.md SHALL carry the top-level `test_case_flow` YAML machine block keyed by tc_id. visual-diff-nav.json SHALL migrate to schema 2.0 per the visual-diff delta with a migration/candidate-generation command; migration notes land in MIGRATION.md.

Enforcement: `harness/scripts/utils/spec-loader.ts`, `harness/scripts/utils/derived-hylyre-plan.ts`, `profiles/hmos-app/harness/visual-diff-nav.ts`

#### Scenario: malformed integration_points are shape-gated not crashed

- **WHEN** integration_points entries are scalars or miss consumer_module
- **THEN** the loader SHALL cull invalid entries with shape_issues provenance feeding `feature_spec_shape`, mirroring the modules[] boundary behavior
