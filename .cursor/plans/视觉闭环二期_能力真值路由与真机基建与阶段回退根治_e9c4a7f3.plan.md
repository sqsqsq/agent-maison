### P0-A 视觉能力真值化：分级信任 + 两状态分离 + 参考图验读 provenance

**1. 分级信任模型（capability_scope）**——替代"新鲜 canary=可信"的单级语义：

```yaml
vision_capability:
  verdict: tool_read | native | none | unknown
  scope: adapter_declared   # 仅声明可能支持——只决定是否值得跑 canary
       | run_probed         # 同 run 内另一 invocation 实测过——只授权"尝试视觉路径"
       | invocation_bound   # 本 invocation 内实测 / 有可信模型路由绑定
```

采信规则：`pixel_1to1` + `vl_multimodal` 终签**只认 invocation_bound**；
`run_probed` 最多授权尝试（prompt 可注 Vision: TRY，产物仍须 provenance 验证）；
`adapter_declared` 与 `unknown` 保守走盲档。cursor 等无法上报模型标识的
adapter：canary receipt 增记 `provider/model/native_image_input/
image_tool_available`（能报则报，报不了记 `model: unknown`），`unknown` 恒
不超过 `run_probed`。

**invocation_bound 签发协议（v3，钉死"谁、凭什么证据、何时签"）**——
签发者恒为 **runner**（信任根），依据二选一，agent 自报一律无效：

- **路径 A（可信模型路由绑定）**：本 invocation 的 adapter/provider/model/CLI
  args（由 runner 拉起参数或 CLI 结构化事件证明，非 agent 自报）与 canary
  receipt 的同名字段 + invocation fingerprint **全等** → 签发。模型标识不可
  证明（cursor auto 路由等）→ 路径 A 不可用；
- **路径 B（同 invocation 内嵌 canary）**：本次 phase invocation 内依次完成
  随机视觉 challenge（runner 出题）→ runner 判卷 → authoritative refs 验读
  （结构化工具事件）→ 业务产出——四段同 invoke_id 绑定 → 签发。
  两路径皆不满足 → 恒 `run_probed`。

**跨阶段 SSOT（v3，能力真值闭环的核心；v4 补显式上下文与优先级冻结）**：
能力与 attestation 状态由 runner-owned receipt 持久化（append-only event +
artifact hash + invoke_id + policy downgrade event），并提供**唯一解析器**：

```ts
resolveEffectiveVisionContext({
  projectRoot, feature,
  runId,          // 隔离跨 run 证据（上一 run 的 downgrade/probe 不串台）
  phase,
  invokeId,       // invocation_bound 只对绑定 invoke 有效
  artifactHashes, // contradiction 只在 artifact hash 相同时继承
})
```

**三轴分算 + fail-closed meet 合成（v5——能力/产物证明/策略降级不是同一
维度，不得用单一优先级链互相覆盖）**。解析器返回结构化三轴，不返回全序
覆盖得出的单一 verdict：

```ts
{
  visionCapability,    // 只由路由绑定/canary/invoke proof 决定（证据新鲜度内取最强）
  artifactAttestation, // 按 artifact hash 逐产物决定（contradicted/unverified/verified）
  effectivePolicy,     // 对全部降级原因做 fail-closed meet（取最保守）
}
```

核心规则：`invocation_bound` **只能提升 visionCapability**，不能覆盖
contradicted/unverified 对当前 artifact 的限制（模型有视觉能力 ≠ 当前产物
可信 ≠ 可解除安全降级——否则绕回本次事故根因：被证伪的 ui-spec hash H 在
后续 invocation 获得 bound receipt 后重新走非盲终签）；blind-safe downgrade
只能由 **runner 显式 supersede/clear 事件**或**符合协议的新 artifact
attestation**（新 hash 重新验证）解除。轴内证据规则：`run_probed` 不跨
run；contradiction 仅同 hash 继承；`invocation_bound` 仅对绑定 invoke 有效。
prompt 注入、spec/coding/testing 各 gate、盲档 kit 派生**只准调用它**取
三轴上下文；禁止一部分消费面读 framework.local.json、另一部分直读
`ui-spec.verified`（否则 spec 判 contradicted 后 plan/coding 仍从 ui-spec
读 verified，blind-safe 降级不生效）。spec retry / 下游阶段一律经该解析器
看到降级态。

**2. 参考图验读 provenance（终签的证据面）**——复用 critic-receipt-producer
既有机制（结构化工具事件→receipt+runner attestation），扩展到 spec 期
authoritative refs：`vl_multimodal` 终签必须携带

- authoritative reference 文件 hash 清单 + 每张的验读工具事件（invoke id 绑定）；
- 未读清单非空 → 不得签 verified（如实 unverified）；
- 无解析器的 adapter（当前除 claude 外）→ 生产者不产出 → **结构性不可签
  vl_multimodal**（诚实降级，盘点合格后逐 adapter 注册解析器，沿用
  adapter-tool-event-provenance.md 流程）。

canary 只证"能看测试图"，不证"读过本需求参考图"——两者在 check 侧分别校验。

**3. 两类状态分离（能力 vs 产物证明——非"attestation 只有两态"；
attestation 本身三 verdict：verified/contradicted/unverified）**——
产出反证推翻的是 artifact，不是模型能力：

```yaml
artifact_visual_attestation:      # 逐产物（ui-spec 等）
  verdict: verified | contradicted | unverified
  reasons: [...]
```

`contradicted` 的效果 = 该产物 `verified` 无效化 + **effective policy 降为
blind-safe**（本 run 视觉路径按盲档纪律执行：kit 强制声明 + 确定性反馈 +
`vl_multimodal` 拒收），措辞与事件明确是"策略保守降级"，**不声称已证明模型
无视觉能力**；`vision_capability` 仅由探针/工具事件决定，不由产物质量反推。

**4. 反证器分级（新检查 `vision_output_counterevidence`，spec 阶段）**：

扫描面为**精确字段**（v3 修正：`must_have_elements` 是元素 **id** 列表非文本，
只扫它一个乱码也抓不到）：`componentNode.text`、`global_elements[].texts`、
text-bearing 节点的 `source_ref`、ref-elements 中与节点绑定的 OCR provenance。

**反证 vs 缺证三态分离（v4——审计语义不得混淆）**：

- **contradicted（已证明矛盾，BLOCKER）**：声明引用图片 hash 与实际文件
  不符；U+FFFD/非法代理对；文本与**高置信** reference 证据明确冲突；
  声明已读但工具事件证明未读；
- **unverified / evidence_gap（证据不足，非"已证伪"）**：OCR 低置信文本
  入 UI（OCR 自己不确定 ≠ UI 文本一定错）；text 无 source/reference 映射
  （缺证明 ≠ 已证明错误）；provenance 缺失；adapter 无事件解析器；
- 两态**同样**使 `vl_multimodal` 失效、同样可安全降 blind-safe——差别只在
  审计记录与报错措辞（"已证伪" vs "证据不足"），不得互相冒充。
- **启发式（首版 observe-only：WARN + 计数落盘，不降档不 BLOCKER）**：
  字典外比率、单字符碎片比例、品牌词不在词典、双通道语义不一致——中文/
  品牌名/银行缩写天然易误伤，先积累两轮真实运行数据再定阈值升级（对齐
  P0-B calibrate→enforce 两段式先例）。

`artifact_visual_attestation` 落**独立 runner-owned receipt**（ui-spec schema
顶层 `additionalProperties:false`，不塞既有产物；ui-spec 只保留既有
verified 字段，其可信度判定走 SSOT 解析器）。

### P0-D 阶段回退编排：改码分类决策表 + 持久化回退状态机

**1. runner 级 source drift reconciliation**（上移，不只 testing check）：
review 闭环后的每个可变阶段（ut/testing）phase 结束时，runner 统一对账产品
源码 drift（复用 attestation 的 diff 面，产出结构化 `changed_files` +
per-file 归因）。

**2. 改码分类决策表（先分类后动作，禁止无差别自动回退——否则 testing agent
可用越权改码触发回退让下游"洗白"）**：

| 变更归因 | 动作 |
|---------|------|
| 本 run 本 phase 授权范围内（phase 规则允许的产物/测试资产） | 不触发 |
| UT seam mutation 且命中**可信授权链**（见下） | 自动回退 + 增量重点复审清单注入 review |
| testing 期产品改码（未授权） | **HALT 求人**（不自动洗白；人工裁决后可显式授权回退） |
| out-of-scope 模块改码 | **HALT** / 走 scope expansion 决议 |
| 外部并发/归因不明（git 状态与 run 记录对不上） | **HALT**（避免覆盖用户修改） |

**approved_src_mutations 可信授权链（v3，钉死信任来源）**——本次事故中
`approved_by: headless-testability-setter-seam` / `headless-testing-integration`
均为 **agent 自签**，不构成授权。"已批准"只认三源：

1. 真人 confirmation receipt（既有 registry 交互确认链）；
2. runner 预定义、范围严格的安全 policy（framework 侧白名单式声明，
   如"仅允许 test 目录内 seam"）；
3. goal 启动前 manifest 明确授权的路径/变更类型。

**拒收**：agent 写入的 `approved_by` 字段、`user_requirement` 泛化哨兵、
headless agent 自产 gap-notes、无文件范围的宽授权（"允许测试 seam"）。

授权 receipt 最少绑定：`run_id / phase / allowed_files / allowed_change_kind /
max_files / source_inventory_before / approved_by / authority_kind
(human | runner_policy | pre_run_manifest) / authority_ref /
receipt_hash`。**pre_run_manifest 源的启动时冻结（v5）**：runner 在
`run_started` 事件冻结 manifest hash（`manifest_hash_at_run_start` +
`manifest_entry_id` 入 receipt）；后续授权判定只引用该快照——运行中被
agent 补写的可变 manifest 不构成授权依据（防"三源"的第三源被运行期
篡改冒充启动前授权）。实际 diff 超出 allowed_files、max_files 或
change kind → 归类翻转为 **unauthorized → HALT**，不自动 backtrack。

**3. 持久化回退状态机**（现 runner 为线性 `for phaseIdx++`，回退必须落
events 不吃内存态）：

- 事件集：`phase_invalidated`（记被失效 phase + attempt + receipt/snapshot
  引用）→ `phase_backtrack_requested`（记分类依据）→
  `phase_backtrack_started` / `phase_backtrack_completed`；
- 被失效 attempt 的 summary/receipt 标记 invalidated（goal report 选取
  最新**有效** attempt，旧 PASS 从 `upstream_verdict_gate` 的新鲜裁决集
  排除——消费面按 invalidation 事件过滤，不删档）；
- **invalidation 消费者矩阵（v4，S1 逐一定案，缺一处=失效不彻底）**：
  `upstream_verdict_gate` / `collectCleanPassIssues` / feature completion
  生成 / upstream closure preflight / `--resume` 起点推导 / goal report
  outcomes / progress.json phase 状态 / receipt & snapshot 最新有效版本
  选择 / review closure freshness / ut & testing summary 读取——全部只能
  看到最新**有效** attempt；验收含端到端断言（失效后逐消费者验证无旧
  PASS 泄漏）；
- `--resume` 从 events 重建回退状态与计数（**进程重启不清零**）；回退上限
  1 次（超限 halt 求人），回退消耗 total turns 与 wall budget（防以回退
  绕预算）；
- 环境层标注（保留 v1 设计）：ut 重跑失败分类为环境（`device_locked` 等）
  → 常驻 summary 附 `failure_layer: environment`，`upstream_verdict_gate`
  文案给"修环境重跑 ut"精确指引（仍拦截不降门禁）。

### P0-E visual ledger 单写者协议（中间轮根治）

**主路线（根治）——goal 态单写者 + 逻辑历史 + 最终重放（v3 补全）**：

- goal 态下 agent 侧 harness **不再直写**正式 ledger，中间轮写
  `intermediate-rounds.journal.jsonl` proposal；
- **逻辑历史（保住 no-progress 语义）**：`evaluateVisualRound` 现只读正式
  ledger——单写者下第 2 中间轮会看不到第 1 轮、熔断语义随孤儿行一起消失。
  goal 态评估历史改为 **committed ledger 基线 + 当前 invoke 已有 journal
  proposals** 拼接视图（只拼本 invoke_id 的 proposal，不吞他人）；
- **runner 收编=顺序重放而非验签（不信任 journal 自带结论）**：从正式
  ledger 基线开始，按 journal `sequence` 逐条重放，**重新计算**每轮
  base_state_hash / decision / row_hash——journal 的 decision/fused 字段
  仅作对照；重算结果与 proposal 全一致 → 写正式 ledger 行 + event；任一
  不一致 → halt（journal 被篡改/评估器漂移都不得静默收编）；
- journal 行 schema：`schema_version / invoke_id / sequence /
  previous_proposal_hash / proposal_hash / source_fingerprint /
  build_fingerprint / screens_hash / gate_fingerprint / structured round
  input（完整输入面，非仅最终 row）`——hash 链**可检测非尾部删行、插行、
  乱序与内容修改；尾部截断属非密码学边界**（删末尾行无后继 hash 暴露
  断链），若需检测须由 runner 文件观察器/外部 head checkpoint/IPC broker
  锚定——规格如实声明该边界，不高估证明力；
- 交互态（非 goal）维持现状直写（无 events 对账面）。

**过渡加固（若单写者需分批落地，先行小修止血）**：收养条件从"pending
attempt"扩展必须同时满足：invoke_id 精确绑定（不只 attempt 号）、
source/build/gate fingerprint 三链一致、行序严格单调、无删改历史行、
每 invocation 中间轮数上限、每条收养行立即写独立 recovery event、
resume 后不重复收养。**跨 attempt 行/历史行改动/删除仍熔断**。

**规格同步**：本项修改 `critic-loop-hardening` 既有 OpenSpec 规格——delta
在 P0-0 change 中显式声明，不许只改代码。

**回归**：本次形态（agent 中途 harness 行→gate 行→下轮对账）不熔断；
伪造旧行/改行/**非尾部删行**/复制行/跨 attempt 行仍熔断（尾部截断属
上文已声明的非密码学边界，不列入熔断承诺——若引入 runner head
checkpoint/IPC broker 锚定再升级为任意删行熔断）。

### P1-I 真机用例独立化与级联归类（硬边界 + 机器 SSOT）

1. **机器 SSOT 落点（v3）**：现状 test-plan.md 为 Markdown 表、前置条件
   自由文本——结构化 DAG **不塞 table cell**，落 test-plan.md 顶层独立
   YAML machine block（以 `tc_id` 为 key）；门禁校验 machine block 与
   Markdown TC 集**完全一致**（缺/多/漂移 → FAIL，防人审表与执行 DAG
   双 SSOT 分叉）：

```yaml
test_case_flow:
  TC-003: { precondition: { kind: after, tc: TC-002 } }
  TC-004: { precondition: { kind: after, tc: TC-003 } }
  TC-009: { precondition: { kind: fresh_app, reset: restart } }
```

2. **DAG 语义细则（v3）**：`after` 支持直接与传递 blocker（TC-003 挂 →
   TC-004 直接 BLOCKED、TC-005 传递 BLOCKED，均记根因 TC）；一个 TC 可依赖
   多个前置（数组形式，任一失败即 BLOCKED）；`after` 引用校验：不存在的
   TC/环引用/传递链断裂 → 派生期 FAIL；
3. **reset 语义**：`fresh_app` 三态命令化（restart=杀进程重拉 /
   clear_data=清应用数据 / fixture_reset=测试夹具重置）；reset 命令本身
   失败 → 归类 **environment failure**（非 root product failure，走
   toolchain 分类），该 TC 记 BLOCKED_BY_ENV；
4. 执行器：`after` 链上前置失败 → 该 TC 记 `BLOCKED_BY <tc>`；
5. **语义硬边界**：BLOCKED_BY **不是 PASS**——仍计入 P0 总分母、仍阻
   feature completion、`device_test_run` 仍 FAIL；唯一变化是根因归类
   （root-fail / blocked-by / independent-fail 三分），报告不再把级联
   显示为 N 个独立缺陷。
