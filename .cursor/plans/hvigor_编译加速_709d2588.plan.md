# hvigor 编译加速 + product 自动探测

详细设计已落到 [.cursor/plans/hvigor_编译加速_a9c1f2e3.plan.md](.cursor/plans/hvigor_编译加速_a9c1f2e3.plan.md)（194 行）。本面板只列要点。

## 核心改动（[framework/harness/scripts/utils/hvigor-runner.ts](framework/harness/scripts/utils/hvigor-runner.ts)）

| 函数 | 现有 args | 新 args |
|------|----------|--------|
| `runHvigorAssembleApp`（coding） | `--mode project -p product=default --no-daemon assembleApp` | `--mode project -p product=<auto> -p buildMode=debug --no-daemon --parallel --incremental assembleApp` |
| `runHvigorBuild`（ut 单模块） | `-p module=<X>@<target> -p product=default --no-daemon <task>` | `-p module=<X>@<target> -p product=<auto> --no-daemon --parallel --incremental <task>` |

**关键证据**：HarmonyOS 官方文档明示 —— HAP/HAR/HSP task 默认 buildMode=debug，但 **App Pack（即 `assembleApp`）默认 release**。这是当前 coding 慢的最大原因（在做 obfuscation/minify/签名优化）。framework 只验"能否过编译 + 出 ohosTest HAP"，不需要发布产物。

## product 自动探测（新增 `detectProduct(projectRoot)`）

回退优先级：
1. `framework.config.json` → `toolchain.preferredProduct`（用户显式覆盖）
2. `build-profile.json5` → `app.products[0].name`（手写 JSON5 容错：去注释 + 去尾逗号 + JSON.parse，不引新依赖）
3. fallback `'default'`（让 hvigor 自己报 product not found，不抢报错）

修了一个隐藏 BLOCKER：内网某工程 product 名是 `mirror`，当前写死 `default` 会直接编译失败。

## 故意不做（避免 v1 过度设计）

- `--mode module` 替代 `assembleApp`：违反 coding 阶段「全工程编译验证」设计意图
- `-p requiredDeviceType=phone`：多端工程会丢模块
- `--analyze=advanced`：诊断开关，反而拖慢
- `--daemon`：脚本一次性任务无 daemon 复用窗口
- `compileFlavor: release` escape hatch：v2 议题，避免给弱模型多一个幻觉旁路

## 验收

1. `npm test` 33+ unit + 11+ fixtures 全 PASS（含本 plan 新增）
2. 本实例工程 `--phase coding --feature <某 feature>` 实测时长降 ≥ 30%
3. 临时构造 `products: [{name: "altproduct"}]` 工程能装配出 `-p product=altproduct`
4. `framework.config.json.toolchain.preferredProduct` 能覆盖探测结果

## 回滚策略

参数粒度回滚（不需要 revert 整个 commit）：
- 移 `--parallel` 留其他 → 移 `--incremental` 留 `buildMode=debug` → 兜底只保留 product 探测（纯 bug 修复，零风险）
