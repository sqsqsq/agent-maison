# 视觉残差闭环（M3 多模态注入 + M4 设备渲染回环）

> 版本窗口：绑定当前在研 `package.json.version = 2.4.0`，**不 bump**。本 plan 是「视觉保真管线升级」（`视觉保真管线升级_7e367678.plan.md`）M3/M4 的真实执行补全，不改原 plan 正文。

## 1. 背景：原 plan 留下的两块「骨架已立、执行未通」

「视觉保真管线升级」按里程碑切分（M1 确定性门禁 / M2 离线量化 / M3 多模态层 / M4 设备回环）。M1/M2 与 M3/M4 的**文档与门禁骨架**均已落地并通过 `npm test`，但两次 review 暴露：M3/M4 的**真实执行链路**只做到「占位/誊写」层级，未真正闭合视觉残差。

### M3 现状（多模态消费侧）

- `harness-runner.collectContextFiles` 已能产出 `ContextFileEntry{kind:image, imagePath, mime}`；`report-generator` 把图片复制到 `reports/.../context-images/` sidecar，并在 `ai-prompt.md` 写 markdown 图片链接。
- `agents/*/adapter.yaml` 已有 `multimodal: boolean`；`multimodal-probe.ts` 已能探测并降级。
- **缺口**：harness-runner **不调用模型 API**，只产出 `ai-prompt.md`，由 verifier（Cursor composer / external_runner headless）消费。sidecar + markdown 链接**不等于**模型真看到像素：弱模型会把图当文本；之前一度往 prompt 内嵌 base64 更是导致 context 膨胀且仍非真附件（已撤回）。**没有任何机制核验 verifier 是否真读了图**。

### M4 现状（生产侧渲染回环）

- `device-testing` SKILL Step 4.6 + `visual-diff-check.ts` 已落地：约定 agent 跑 Hylyre 截图、多模态对照原图、**手填** `visual-diff.json`，harness 校验 JSON schema（已加 `[0,1]` 域校验、`ref_id` 对照、截图 `existsSync`）。
- **缺口**：截图与对照**全靠 agent 手工誊写**；harness 既不自动截图，也无任何客观分；`fidelity_score` 完全由填写者给定。渲染→diff→回修闭环**未自动化**，残差闭合依赖人自觉。

> 归属澄清：两块在原 plan 中**均有方向**（M3=§7.I 图片注入 + adapter `multimodal` 位；M4=§4.F 设备回环、SKILL Step 4.6）。本 plan **不新增范围**，只把「骨架」推进到「可执行 + 可核验 + 可降级」。

## 2. 目标

| 里程碑 | 目标 | 验收信号 |
|--------|------|----------|
| **M3** | 图片注入按 adapter 能力**分级真消费**，并对 verifier「是否真看图」留**可核验证据**，缺证据则显式降级 | tool_read adapter 跑 verify 时产出「已读图证据」块；缺证据 → `visual_multimodal_parity=WARN` 标降级；`adapter image_input=none` → 文本降级，文案区分 |

## 3. M3 设计：按 adapter 能力分级的真多模态注入

### 3.1 能力分级（adapter 契约）

`agents/adapter-schema.yaml` 的 `multimodal: boolean` 升级为三档 `image_input`：

| 取值 | 含义 | 第一版 adapter |
|------|------|----------------|
| `none` | 模型不支持图像；降级到文本 ui-spec + 确定性 parity | generic（内网盲模型管道）/ 纯文本内网模型 |
| `tool_read` | 模型/agent 可用**读图工具**读取本地 sidecar 文件（Cursor composer、claude Read） | cursor、claude |
| `native_attach` | 通过 SDK/CLI 把图片作为**原生附件**多模态输入 | 预留（goal-mode headless 未来路径，见 §3.5） |

**向后兼容**：仍接受旧 `multimodal: true|false`，运行时映射 `true→tool_read / false→none` 并 stderr 一次性弃用提示；现有 `multimodal` 单测不破坏。

**须区分交互 vs headless 维度**：同一 adapter 的视觉能力在交互态与 goal/headless 态**可能不同**，`image_input` 声明须与 `goal_capability` 对齐（见 §3.5）。例如 claude 交互 composer 与 `claude -p --allowedTools ...,Read,...` 两态都带读图工具 → 两态 `tool_read`；generic 管道接内网盲模型 → headless 实为 `none`。声明时若两态不一致须显式标注，避免「交互能看图」被误推广到 goal 态。

### 3.2 消费路径

- **tool_read（主路径）**：`report-generator` / `verify-coding.md` 在 prompt 中显式指令 verifier「**用读图工具逐个读取 `reports/.../context-images/` 下 sidecar**，并在结论中列出**已读图证据**：读了哪些文件 + 每图关键观察（版面/品牌色/资产/文案）」。
- **native_attach（slim 预留）**：仅在 schema 保留枚举 + 文档说明回退语义；**第一版不写 `agent-invoke.ts` 通道代码、不写无人走的回退测试**。**定位（重要）**：native_attach 的真正归宿是 **goal-mode headless 多模态补齐路径**（见 §3.5）——交互态 Cursor 有 tool_read 即够，headless 态若某 CLI/SDK 支持图片附件参数，native_attach 是不依赖交互式读图工具、直接喂像素的通道。当前无任何 adapter 的 headless CLI 确认支持附件参数，故 v1 slim（写通道=两端没接东西的水管）；一旦出现支持附件的 headless CLI，即在此接入。
- **none**：维持现有文本降级 + 「视觉多模态层已降级」标注。

### 3.3 读图证据块契约（① 最高优先，软门禁的前提）

软门禁要解析的「证据」来自 verifier 的自由文本（harness 不调模型）。若不先钉死一个**确定性可机读 marker**，解析就是正则猜谜、必然误判。故定义**唯一契约 SSOT**：

```read-image-evidence
- file: 00-1-银行卡添卡首页.jpg
  observation: 顶部红色品牌条 + 整行储蓄卡/信用卡卡片
- file: 01-3-选卡半模态.jpg
  observation: 储蓄卡/信用卡各占整行 + 右侧勾选
```

（或等价的固定锚 `## 已读图证据` + 文件名勾选清单——二选一定型后唯一化。）

**关键约束**：该契约由 **prompt 产出端（§3.2 / M3-3b）** 与 **gate 解析端（§3.4 / M3-5）** 共用同一份 SSOT（落在 `verify-coding.md` 正文 + 一个共享解析 util），禁止两端各写各的正则导致漂移。

### 3.4 证据软门禁

`visual_multimodal_parity` 检查（按 §3.3 契约解析）：

- verifier 输出含读图证据块 → 正常评估；
- tool_read 但**无证据块** → **WARN**，details 写「未取得读图证据，多模态降级（区别于 adapter 不支持）」；
- adapter=none → SKIP/WARN 标「adapter 不支持图像」。

仅**软门禁**（MAJOR/WARN），不阻断 compile/其它闭环——延续原 plan「M3 不阻塞 M1」。

**跨 adapter 落点（实现前必须钉死；claude-scoped 诚实版）**：`visual_multimodal_parity` 是 `verify-coding.md` 七-b 节里由 VL verifier 自填的 prompt 级检查项（harness 不调模型）。**强制解析的落点是 verify 报告产物（文件），不是 raw stdout**：

- **Claude（交互 + goal 两态）**：verifier 子 agent 的 transcript 经 SubagentStop hook（`agents/claude/templates/hooks/record-verifier-report.mjs`）落成 `verifier.report.md`（供 `check-receipt.ts` 经 `verifier_subagent.report_path` 引用）。M3-5 **解析该文件**抠 §3.3 证据块 → 强制核验。
- **非 Claude（Cursor 交互 + chrys/opencode/codex/generic 的 goal）**：既无 verifier 子 agent、也无 `verifier.report.md` 产出者；goal-runner 捕获的 agent stdout 只落 `agent-output.log`（审计/活性，**无人解析出 verdict**——verdict 由 `summary.json` + harness exit 裁决，见 `goal-runner.ts:1008` `resolvePhaseHarnessVerdict`）。故这些 adapter **标 prompt 自律 / 无强制解析**，不假装 goal 态能强制兜。

**诚实原则**：plan 不得宣称「harness 普遍强制核验读图证据」——只有 Claude 经 hook 文件强制；其余靠 prompt 指示 + 文本降级。文案区分「Claude 强制解析 / 非 Claude 自律 / adapter=none 降级」三态。真要 adapter 无关强制解析（扫 `agent-output.log` marker）是未来增量，非本 plan 范围。

### 3.5 Goal/headless 模式视觉能力与降级

> 动机（用户提问 2026-06-23）：goal 模式经 `external_runner.headless_invoke` spawn headless agent，大部分内网 CLI 无读图能力，是否导致 goal 模式视觉能力弱于交互模式？

**结论：弱化不来自 goal 模式本身，而来自「模型/CLI 能力」这一维**。同一内网盲模型在交互模式下也看不到图；而具备读图工具的强 agent 走 headless 仍能 tool_read。

视觉能力矩阵（adapter × 形态；structured adapter 清单见 `agent-invoke.ts:33` `KNOWN_STRUCTURED_ADAPTERS = {claude, codex, cursor, chrys, opencode}`）：

- **claude**：交互 composer 与 `claude -p ... --allowedTools Bash,Read,...`（`agents/claude/adapter.yaml:67`，Read 能读图、sidecar 在工程内可达）两态均 **tool_read**，goal 模式**不弱**。
- **codex**：`codex exec`（`agents/codex/adapter.yaml:32`）是否读图取决于该 CLI 是否保留读图/多模态——须实测定档。
- **cursor**：交互 composer = tool_read；goal 走 cursor-agent CLI 时按其读图能力定档。
- **chrys**：`chrys run --task {{PROMPT_FILE}} -C {{PROJECT_ROOT}} --agent Code --json`（`agents/chrys/adapter.yaml:28`，external_runner）——**goal 宿主之一，须显式定档**（大概率 headless 实为 `none`，按实测）。不得漏在矩阵外。
- **opencode**：external_runner（stdin）——同 chrys 须显式定档（大概率 `none`，按实测）。
- **generic**：`{{PROMPT}} | agent-cli -`（`agents/generic/adapter.yaml:29`）接内网盲模型 → headless 实为 **none**。**且是模型问题非 goal 问题**。

**补齐 goal/headless 视觉能力（四层，原 plan 架构已为此设计）**：

1. **文本 ui-spec 解耦（主补齐，原 plan J）**：强 VL 在 spec 阶段把视觉信息抽成纯文本 ui-spec（组件树+token+每品牌色+逐字文案+资产 key）；goal 下游盲模型**消费文本**即拿到绝大部分视觉信息，无需自看图。
2. **确定性门禁满血（D/K，模型无关）**：`check-visual-parity`、静态保真分在 goal 模式不打折。
3. **native_attach（goal-mode headless 未来路径）**：支持附件参数的 headless CLI 出现时，是不依赖交互读图工具的喂像素通道（v1 slim 预留，见 §3.2）。
4. **证据软门禁（M3-5）保证降级诚实**：goal 模式盲模型 → `visual_multimodal_parity=WARN`「未取得读图证据，多模态降级」，诚实可见不伪绿。

**证据强制解析的落点 = 文件，非 stdout（见 §3.4）**：goal-runner 捕获的 agent stdout 只落 `agent-output.log`（审计/活性），**无人解析出 verdict 或证据**（verdict 由 `summary.json` + harness exit 裁决）。故读图证据**强制**核验仅对 Claude 成立（经 SubagentStop hook 落 `verifier.report.md` 文件）；非 Claude（含 chrys/opencode/codex/generic 的 goal）= prompt 自律。**goal 态并不比交互态在证据门禁上更强**（goal-runner 不解析 agent stdout）。

**tool_read 的 goal footgun（preflight 须断言）**：`claudeArgv` 默认工具集含 Read，但 `unattended.allowed_tools`（`goal-manifest.ts:26`）一旦传入即覆盖默认（`agent-invoke.ts:224-235`）。若某 manifest 把 `allowed_tools` 收窄成 `Bash,Edit`，`tool_read` 会**静默失效**（交互态无此开关）。故 **`image_input=tool_read` 的 goal adapter，goal-runner preflight 须断言有效 `allowed_tools` 含 Read（或等价读图工具），否则诚实降级 `none`**——而非声称 tool_read 却无读图工具。

**拓扑建议（K2）**：吃图的 spec 提取钉给强 VL（交互产出或带读图工具的 goal agent），编码交 goal 弱模型消费文本 ui-spec；**禁用盲模型跑 spec 提取**（garbage in，原 plan J 反模式）。

## 4. M4 设计：harness 复用 Hylyre 的自动截图回环

### 4.1 自动截图采集

新增 `profiles/hmos-app/harness/visual-diff-capture.ts`：

1. 复用 `providers/device-test-run` + `hylyre-spawn` 的 `screenshot` 能力；
2. 按 ui-spec **P0 且可直达顶层屏**自动导航 + 截图，落 `device-testing/device-screenshots/shot-<screen>.png`；深层屏复用 device-testing 导航或标 `pending`（MVP 与原 plan §4.F 一致，先覆盖可直达屏）；
3. 生成 `visual-diff.json` **骨架**：`{screen_id, screenshot_path, ref_id, verdict: "pending"}`。

**架构归属（遵守既有「运行 vs 校验」分层）**：截图采集是**带设备副作用的运行时动作**，归 `device_test.run` 运行流程（与 `runHylyreDeviceTest` / `ensureHylyreReady` 同层），**不得**塞进 check-testing 的确定性校验期——`dispatchDeviceVisualDiff` 返回 `CheckResult[]` 且应保持无副作用。复用既有 provider `profiles/hmos-app/harness/providers/device-test-visual-diff.ts`（当前仅 `export { checkVisualDiff }`），在其 `exports` 上扩展采集函数（如 `captureVisualDiff`），避免另开第二个 `visual_diff` 入口造成双源。

### 4.2 QA action 区分（呼应原 plan §4.F）

`visual_diff` 截图是**阶段级 QA/门禁动作**（harness 直接发起），**不是** test-plan 派生步骤根键。须在文档显式区分，并（若实现需经 Hylyre 步骤通道）为 `visual_diff` 登记一个**允许的 QA action**，而非放开通用 `screenshot` 派生步骤根键——与 `hylyre-planned-step-fields.md` 现有约束不冲突。

### 4.3 客观下限分（可选 / 半定量）

用 `image-toolkit`(jimp，M2 已选型) 对 **device shot vs authoritative ref** 做**尺寸归一后**的结构/直方图相似度，产出 `score_floor`（写入 `visual-diff.json`）：

- **明确半定量**：非 CLIP、非语义；只作 `fidelity_score` 的**客观下限/异常哨兵**（例如 floor 极低却被填高分 → 提示复核）；
- VL 仍出最终 `verdict` 与 `must-fix`；
- 设备/jimp 不可用 → 跳过该字段，不阻断。

**与既有字段语义分工（避免混淆，须文档讲清）**：`visual-diff-check.ts` 已有两个 **VL 自填**字段——`fidelity_score`（语义保真分，受 `PASS_MIN_FIDELITY=0.6` 阈值约束）与 `geometric_iou`（几何，受 `PASS_MIN_IOU=0.5` 约束）。新增 `score_floor` 是 **jimp 客观下限/哨兵**，**不参与现有 PASS 阈值判定**。三者职责不同、不可混用。

**哨兵交叉软校验**：引入 `score_floor` 的唯一价值在于一条交叉 check——`M4-4` 须实现：当 `score_floor` 存在且 `fidelity_score` 显著高于 `score_floor`（差距超阈值）→ 出 **WARN**「VL 高分但客观相似度低，人工复核」。这能兜住「VL 自报高分偷懒」：例如银行卡渲染成全局蓝（原图招商红），VL 写 `fidelity_score: 0.85` 而 jimp 直方图 `score_floor=0.3` → WARN 提示漏检品牌色。**软 WARN 不硬 FAIL**（floor 半定量、噪声大，仅供人工复核）。**若不实现此交叉校验，`score_floor` 即死重**——届时应重新掂量 M4-3 是否值得做。

> 设计取舍：把客观算分定为**可选半定量**而非硬指标，避免 DPI/字体/动态内容导致的「算不准反污染 trace」（与原 plan §4.K 对静态几何 IoU 的顾虑同源）。真·几何/像素相似度在此处（渲染后）才有意义，故放 M4 而非静态侧。

### 4.4 校验与降级

- **前置改动（否则骨架一落地即 FAIL）**：现有 `validateVisualDiffJson` 对非 `pass|warn|fail|skipped` 的 verdict 直接 push「verdict 非法」并 FAIL（`visual-diff-check.ts:118-121`）。故须**先**扩 `VisualDiffScreenEntry.verdict` 联合类型 + harness JSON schema 放行 `pending`，**再**令其走 WARN。
- **`pending` 三细节（③）**：
  - **(a) 语义区分**：`pending`=未判（已采集截图但 VL 尚未填 verdict）；`skipped`=主动跳（设备不可用/不在范围）。文案与处理须分开。
  - **(b) 不计入 `effectiveScreens`**：`pending` 屏不算有效对照，全 `pending` → WARN 非 PASS（与现有 `effectiveScreens=0` / P0 未覆盖 WARN 逻辑协同）。
  - **(c) 免填分数**：`pending` 屏免填 `fidelity_score`/`geometric_iou`（现有代码仅在 `verdict=pass|warn` 时要求，**天然满足**，但须加测试固化，防骨架被域校验拦下）。
- **哨兵交叉软校验**：`score_floor` 存在且 `fidelity_score` 显著高于它（差距超阈值）→ **WARN** 复核；软 WARN 不硬 FAIL。`score_floor` 仍受 `[0,1]` 约束，但**不参与** `PASS_MIN_*` 阈值判定。
- warmup/无设备/Hylyre 不可用 → 全程优雅 **SKIP** 标「仅静态保真分生效」，不硬 exit。

## 7. 风险与边界

- **R1 verifier 不可强制**：harness 不调模型，无法强制 verifier 真读图——故采**证据软门禁**（缺证据降级而非伪绿），延续 M3「不阻塞 M1」。
- **R2 客观算分误导**：DPI/字体/动态内容使像素相似度不可靠——定为**可选半定量哨兵**，不作硬指标、不替代 VL。
- **R3 Hylyre 易碎（鸿蒙重型件）**：warmup/无设备频繁失败——全程优雅 SKIP，复用既有降级路径，不硬 exit。
- **R4 与 hylyre 步骤根键冲突**：`visual_diff` 走 QA action 而非派生步骤根键，文档显式区分。
- **R5 adapter schema 变更兼容**：`multimodal:bool` 保留兼容映射 + 弃用提示，分窗口移除。
- **R6 证据软门禁仅覆盖 verify 侧**：M3 读图证据核验只作用于 verify-coding 的 `visual_multimodal_parity`；**coding 侧 Step 2.5「BLOCKER Read 原图」无任何证据核验机制**（无法证明编码模型真读了图）。这是已知边界，不在本 plan 范围内强行补，但文案勿暗示 coding 读图也被核验。
- **R7 采集入口架构错配**：自动截图带设备副作用，若误入 check-testing 确定性校验期会破坏「校验无副作用」约定——故归 `device_test.run` 运行流程，复用既有 `device-test-visual-diff.ts` provider（见 §4.1 / §5）。
- **R8 证据块解析靠自由文本（① 核心风险）**：verifier 输出是自由文本，无确定性 marker 则软门禁=正则猜谜必误判——故 §3.3 先钉死可机读契约 SSOT，prompt 产出端与 gate 解析端共用，是 M3-5 的前置。
- **R9 capture 冷启动可达判定（落地难点）**：「P0 且可直达顶层屏自动导航」要求 capture 侧有「冷启动可达」判定，而 check 侧只有 `collectP0ScreenIds`；深层屏复用 device-testing 导航。与原 plan §4.F MVP 边界一致，但实际导航编排是真正难点，MVP 先覆盖可直达屏、深层标 `pending`。
- **R10 goal/headless 盲模型路径（见 §3.5）**：generic 管道接内网盲模型时 goal 模式视觉层退化为文本——非 goal 模式缺陷而是模型能力问题。靠文本 ui-spec 解耦（J）+ 确定性门禁（D/K，模型无关）+ 证据软门禁（M3-5，诚实降级）+ native_attach 预留（未来）兜底；并建议吃图 spec 提取钉强 VL（K2），禁盲模型跑提取。**证据强制核验仅 Claude**：非 Claude 退 prompt 自律——goal 态在证据门禁上并不强于交互态（详见 §3.4）。
- **R11 chrys/opencode 漏档**：二者在 `KNOWN_STRUCTURED_ADAPTERS` 内且 chrys 是当前 goal 宿主；`image_input` 与 §3.5 矩阵须显式定档（大概率 headless none，按实测），否则正在发版的 adapter 漏在视觉能力矩阵外。
- **R12 tool_read goal footgun**：`allowed_tools` 收窄会让 `tool_read` 静默失效（交互态无此开关）；靠 §3.5 的 preflight 断言（含 Read 否则降 none）兜底。

## 8. 里程碑边界

- **M3** 与 **M4** 各自独立可交付：M3 只动 adapter/prompt/probe + 证据契约（native_attach slim、不动 agent-invoke），不依赖设备；M4 只动 device-testing/visual-diff，不依赖多模态注入。任一里程碑不阻塞另一已交付价值。
- 建议先 **M3**（纯软件、无设备依赖、ROI 高），后 **M4**（依赖 Hylyre/设备，含可选新算分）。
