## 一、改造目标

- **架构级变更**（三类任一命中）才触发 `doc/architecture.md` 更新：
  - `dsl_change`：`framework.config.json > architecture` 结构变化（分层 / 依赖边 / 同层策略 / 内层顺序 / 出口约定）
  - `module_set_change`：模块集合变化（新增 / 下线 / 迁层）
  - `responsibility_rewrite`：`module-catalog.yaml > primary_responsibility` 大幅重写
- **feature 级变更**（既有模块内新增页面 / 接口 / 数据模型 / 修复）**一律不动** `architecture.md`，变更历史由 git + `doc/features/<feature>/` 承担。
- `architecture.md` 保留**极简业务模块清单**（模块名 + 外层 + 一句话 + 链到 `module-catalog.yaml`），仅在 `module_set_change` 时更新。
- 把硬门禁做成"条件式"：design.md 声明 `architecture_impact: none` 时自动 SKIP，声明为其他值时再强校验。

## 二、新增契约：design.md 的"架构影响声明"

在 design.md 新增固定章节（放在 `## 1. Scope 声明与继承` 之后、原模块架构图之前）：

```markdown
## 1.5 架构影响声明 (architecture_impact)

- impact: none | dsl_change | module_set_change | responsibility_rewrite
- affected_items: []            # impact != none 时必填，列出受影响的模块/外层/DSL 段落
- architecture_md_updates: []   # impact != none 时必填，列出 architecture.md 哪些小节要改
- catalog_updates: []           # 可选，列出 module-catalog.yaml 哪些条目要改
```

`impact: none` 是**缺省的、最常见的情况**，作者只需勾选即可；`impact != none` 才需要填写后续字段。
