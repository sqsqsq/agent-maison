# 消费者验证命令系统性重构

## 问题本质

`npm test` 在源仓与消费者工程中有不同语义，但当前实现混为一谈导致消费者 init 必然失败：

矛盾锁链（5 处）：

1. [`release-excludes.json`](scripts/release-excludes.json) L10 — 排除 `harness/tests/**`
2. [`harness/package.json`](harness/package.json) L16 — `"test"` 指向被排除的 `tests/run-unit.ts`
3. [`verify-release-pack.mjs`](scripts/verify-release-pack.mjs) L130 — 断言发布件必须有 `scripts.test`
4. [`SKILL.md`](skills/00-framework-init/SKILL.md) L136 — BLOCKER 要求 `npm test` 通过
5. `harness-runner.ts` L234 — 裸调用无 `--phase` 直接 exit 1（不能简单 alias `test` -> 裸 `check`）

## 设计决策（单一明确策略）

**消费者 `npm test` = `check:global` = catalog + glossary + docs 三个全局 phase 链式执行。**

与 init S3 的 `run-global-phases`（[`init-task-executor.ts`](harness/scripts/utils/init-task-executor.ts) L199-218）行为一致。

理由：
- 每条 `check:*` 携带显式 `--phase`，不触发 harness-runner "必须指定 phase" 报错
- 三个全局 phase 不依赖 `--feature`、不依赖 DevEco/真机，任何消费者都能跑
- 语义精确：验证 framework catalog/glossary/docs 元数据完整性
- 与 S3 实际行为对齐，不引入新的语义坑

### 目标态对照

**发布版 harness/package.json**（sanitize 后）：

```json
{
  "check": "ts-node harness-runner.ts",
  "check:prd": "ts-node harness-runner.ts --phase prd",
  "check:design": "ts-node harness-runner.ts --phase design",
  "check:coding": "ts-node harness-runner.ts --phase coding",
  "check:review": "ts-node harness-runner.ts --phase review",
  "check:ut": "ts-node harness-runner.ts --phase ut",
  "check:testing": "ts-node harness-runner.ts --phase testing",
  "check:catalog": "ts-node harness-runner.ts --phase catalog",
  "check:glossary": "ts-node harness-runner.ts --phase glossary",
  "check:docs": "ts-node harness-runner.ts --phase docs",
  "check:global": "npm run check:catalog && npm run check:glossary && npm run check:docs",
  "test": "npm run check:global"
}
```

**删除**：`test:unit`、`test:fixtures`

**发布版根 package.json**（sanitize 后）：

```json
{
  "harness:install": "npm --prefix harness install",
  "test": "npm --prefix harness test"
}
```

链路：根 `test` → harness `test` → harness `check:global` → catalog && glossary && docs

**源仓不变**：

```json
"test": "npm run test:unit && npm run test:fixtures",
"test:unit": "ts-node tests/run-unit.ts",
"test:fixtures": "ts-node tests/run-tests.ts"
```

## 实施清单

### Phase 1: 源仓 harness/package.json 补充 check:global 系列

[`harness/package.json`](harness/package.json) 新增（不改现有 test）：

```json
"check:catalog": "ts-node harness-runner.ts --phase catalog",
"check:glossary": "ts-node harness-runner.ts --phase glossary",
"check:docs": "ts-node harness-runner.ts --phase docs",
"check:global": "npm run check:catalog && npm run check:glossary && npm run check:docs"
```

源仓可保留该脚本但不作为源仓回归门禁（源仓根下无 `doc/module-catalog.yaml` 等文件，裸跑会 FAIL）。它适合作为"消费者 init 后验证命令"。

### Phase 2: 打包 sanitize

**2.1 [`scripts/release-pack-rules.mjs`](scripts/release-pack-rules.mjs) — 新增 `sanitizeHarnessPackageJson(pkg)`**

```javascript
export function sanitizeHarnessPackageJson(pkg) {
  const out = JSON.parse(JSON.stringify(pkg));
  if (out.scripts) {
    delete out.scripts['test:unit'];
    delete out.scripts['test:fixtures'];
    out.scripts.test = 'npm run check:global';
  }
  return out;
}
```

**2.2 [`scripts/pack-release.mjs`](scripts/pack-release.mjs) — staging 时处理 harness/package.json**

与根 pkg 同模式：读取源仓 `harness/package.json` → `sanitizeHarnessPackageJson` → 写入 staging 目录的 `harness/package.json`（覆盖 collectReleaseFiles 拷贝的原始版本）。不要只 sanitize 根 pkg。

**2.3 根 pkg sanitize 微调**

`sanitizePackageJson`：保留 `scripts.test`（值 `"npm --prefix harness test"` 无需改），删 `openspec` / `openspec:validate`（消费者无此依赖）。

### Phase 3: 验收断言强化

**3.1 [`scripts/verify-release-pack.mjs`](scripts/verify-release-pack.mjs)**

更新 `verifyReleasePack` 中的断言：

- 根 pkg：保留 `scripts.test` + `scripts['harness:install']` 断言
- 新增：读取**解压后**的 `framework/harness/package.json`（即 zip 内 / staging 内的版本，而非源仓的 `harness/package.json`）：
  - `scripts['test:unit']` **不存在**
  - `scripts['test:fixtures']` **不存在**
  - `scripts['check:global']` **存在**
  - `scripts.test` === `'npm run check:global'`

注意：必须读 frameworkRoot 下（解压 zip 后的路径）的 harness/package.json，否则断言永远对着源仓通过，失去防回退意义。

**3.2 `release-pack-rules.mjs` 的 `runSyntheticRuleTests` 扩展**

加入 `sanitizeHarnessPackageJson` 白盒单测（类似现有根 pkg sanitize 测试）。

### Phase 6: 端到端验收

```bash
# 1. 源仓回归
cd harness && npm test   # unit + fixture 全 PASS（不变）

# 2. 打包规则（纯静态断言，不需要网络/npm install）
npm run release:verify   # 新断言全部通过
```

**消费者 staging smoke（可选独立命令，不在 release:verify 主流程内）**：

`check:catalog` / `check:glossary` 对缺失的 `doc/module-catalog.yaml` / `doc/glossary.yaml` 判 BLOCKER FAIL，
因此不能裸跑在空 staging 目录上。正确 smoke 需要构造最小消费者环境：

```bash
# 构造 tmpdir：最小消费者根 + framework/ + doc/ 骨架
npm run release:pack -- --stage-only
# 脚本自动在 tmpdir 中：
#   mkdir -p <tmp>/doc
#   写入最小合法骨架（非空文件，parser 要求 root 为 object）：
#     doc/module-catalog.yaml:  schema_version: "1.0"\nmodules: []
#     doc/glossary.yaml:        schema_version: "1.0"\nterms: []
#     (doc/glossary-seed.txt 不存在 → harness 会 SKIP，非 BLOCKER)
#   cp -r dist/release-staging/framework/ <tmp>/framework/
#   echo '{}' > <tmp>/framework.config.json   (最小合法 config)
#   cd <tmp>/framework/harness && npm install && npm test
# 期望：check:global 三 phase 无 BLOCKER FAIL（catalog/glossary 空骨架 → WARN 可接受）
```

实现方式二选一：
- 新增 `scripts/smoke-consumer-staging.mjs`（独立命令，手动触发）
- 或纳入 `verify-release-pack.mjs` 的可选 `--with-smoke` 参数（默认不跑，避免 npm install 依赖网络）

实现注意：`smoke-consumer-staging.mjs` 是跨平台脚本，用 Node 的 `fs.cpSync` / `fs.mkdirSync` / `fs.writeFileSync` 构造 tmpdir，不要用 shell `cp -r` / `mkdir -p`。

关键：**verify 主流程的两条硬断言**（不依赖 npm install，纯 JSON 读取）：
1. staging `harness/package.json` 不含 `test:unit` / `test:fixtures`
2. staging `harness/package.json` 的 `scripts.test` === `"npm run check:global"`
