# tsc 门禁加固（修 TS2503 + 全 profiles 覆盖 + 双挡 tsc --noEmit）

## 背景（含两轮 review 实测修订）

- 全量 `tsc --noEmit -p harness/tsconfig.json` 当前只报 5 个错，全在 [graph-extractor-host.ts](profiles/hmos-app/harness/graph-extractor-host.ts)（L31/32/59/81/102），均 `TS2503: Cannot find namespace 'ts'`。
- 根因：`const ts = harnessRequire('typescript')` 是运行时**值**，但代码在**类型位置**用了 `ts.CompilerOptions` / `ts.CompilerHost` / `ts.ScriptTarget` / `ts.Node`——值不能当命名空间类型。
- `npm test` 与 `release:verify` 都不跑全量 tsc，所以这类错误一直漏进发布件（goal-runner 那次同源）。
- 同款 `harnessRequire('typescript')` 的 [ts-compile.ts](profiles/hmos-app/harness/ts-compile.ts) 只在值位置用 `ts.ScriptTarget.ES2020` 等，**不报错、不改**。

### 两处实测修订（推翻初版）

- **模块解析（P1）**：实测 `typescript` 只装在 `harness/node_modules`，根目录与 profiles 链上都没有。从 `profiles/hmos-app/harness/` 写裸 `import type ... from 'typescript'` 会解析失败（TS2503 变 TS2307）。**改用相对 type import** 指向真实位置。
- **覆盖面（P2）**：`harness/tsconfig.json` 的 `include: ./**/*.ts` 只覆盖 harness 本体 + import 链拉入的 profile 文件；`ut-host-impl.ts` / `providers/ut-run.ts` / `device-test.ts` / `coding-compile.ts` / `named-handler.ts` 等 **harness-runner 运行时动态 require 的 provider 入口**在静态图外，类型错误照漏。**新建覆盖全 profiles 的 tsconfig.typecheck.json**。

## 改造点 1：修 TS2503（相对 type-only import，运行时零变化）

[graph-extractor-host.ts](profiles/hmos-app/harness/graph-extractor-host.ts) 顶部：

```ts
import type * as TsModule from '../../../harness/node_modules/typescript';
const ts: typeof TsModule = harnessRequire('typescript');
```

- 相对路径与该文件已有的 `path.resolve(__dirname,'..','..','..','harness','package.json')` 布局假设一致（三级上溯到仓根再进 harness），不引入新脆弱点；consumer 布局下 `framework/profiles/hmos-app/harness → framework/harness/node_modules` 同样成立。
- `import type` 编译期被 erase，**不产生运行时 require**，继续走 `harnessRequire`。
- 仅把**类型位置**的 `ts.` 换成 `TsModule.`，值位置保持 `ts.`：
  - L31 `options: ts.CompilerOptions` → `TsModule.CompilerOptions`
  - L32 `): ts.CompilerHost {` → `): TsModule.CompilerHost {`
  - L59 `languageVersion: ts.ScriptTarget,` → `TsModule.ScriptTarget`
  - L81 `const options: ts.CompilerOptions = {` → `TsModule.CompilerOptions`
  - L102 `const visit = (node: ts.Node)` → `TsModule.Node`
- 值位置不动：`ts.createCompilerHost`(L33)、`ts.createSourceFile`(L65)、`ts.ScriptTarget.ES2020`/`ts.ModuleKind`/`ts.ModuleResolutionKind`(L82-84)、`ts.createProgram`(L95)、`ts.isFunctionDeclaration` 等(L103-117)。

## 改造点 2：新建 tsconfig.typecheck.json（全 profiles 覆盖）

新增 `harness/tsconfig.typecheck.json`：

```json
{
  "extends": "./tsconfig.json",
  "compilerOptions": {
    "noEmit": true,
    "declaration": false,
    "declarationMap": false,
    "sourceMap": false
  },
  "include": ["./**/*.ts", "../profiles/**/*.ts"],
  "exclude": [
    "node_modules",
    "dist",
    "**/tests/**",
    "../profiles/**/node_modules/**",
    "../profiles/**/vendor/**"
  ]
}
```

- `extends` 主 [tsconfig.json](harness/tsconfig.json)，但强制 `noEmit` 并关掉 declaration/sourceMap（职责与 build 分离）。
- `include` 扩到 `../profiles/**/*.ts`，把动态 require 的 provider 入口纳入静态检查。
- 排除 `tests`（不进发布件，由 `npm test` 实际运行兜底）、`node_modules`、`dist`、`vendor`。
- **实施第一步先单独跑** `npx tsc --noEmit -p tsconfig.typecheck.json`，确认仅清掉已知 5 个 TS2503、无超预期新增。
- **纪律（Codex review）**：若扩 include 后暴露额外类型错误，**优先修真实类型问题，不得为过门禁而缩窄 include**；仅当确认是 tests / vendor / 非发布内容时才追加精确 `exclude`。

## 改造点 3：harness npm test 串入 typecheck（开发期挡）

[harness/package.json](harness/package.json)：

```json
"typecheck": "tsc --noEmit -p tsconfig.typecheck.json",
"test": "npm run typecheck && npm run test:unit && npm run test:fixtures",
```

- 本地/CI `cd harness && npm test` 会先 typecheck，类型错误即时暴露。

## 改造点 4：release:verify 加 tsc 门禁（发布期挡）

[verify-release-pack.mjs](scripts/verify-release-pack.mjs) `verifyReleasePack()` 开头（synthetic rule tests 之前，fail-fast）加一步：

```js
import { spawnSync } from 'child_process';
// ...
console.log('[release:verify] typecheck (tsc --noEmit)...');
const tc = spawnSync('npm', ['run', 'typecheck'], {
  cwd: path.join(REPO_ROOT, 'harness'),
  stdio: 'inherit',
  shell: true,
});
if (tc.status !== 0) fail('typecheck (tsc --noEmit) failed');
console.log('[release:verify] typecheck PASS');
```

- 复用 harness 的 `typecheck` script，避免命令双写。
- 前置：harness 已 `npm install`（typescript 为 harness devDependency）。
- 片段中的 `REPO_ROOT`(L20) 与 `fail()`(L23) 该文件已有，按其现有风格对齐即可，不引入第二套约定。

## 改造点 5：发布件保留 typecheck（P3 采纳 Claude）

- **不**改 `sanitizeHarnessPackageJson`——保留 `typecheck` script，consumer 可 `cd framework/harness && npm run typecheck` 自诊断（typescript 本就是 harness devDep，consumer 安装即有）。
- 注意 sanitize 仍把发布件 `test` 改写为 `npm run check:global`，故 consumer 跑测试不会因类型 fail，typecheck 是显式可选自查。
- 确保 `harness/tsconfig.typecheck.json` **进发布件**：核对 [release-excludes.json](scripts/release-excludes.json) 未排除它（它 include `../profiles/**`，发布件 `framework/harness` 与 `framework/profiles` 相对关系成立）。
- [verify-release-pack.mjs](scripts/verify-release-pack.mjs) `assertZipContents()` 补断言（反转原"删除"断言）：
  - 发布件 `harness/package.json` 仍含 `typecheck` script；
  - 发布件存在 `harness/tsconfig.typecheck.json`。

## 验收

```bash
cd harness && npx tsc --noEmit -p tsconfig.typecheck.json   # 0 error（全 profiles 覆盖）
cd harness && npm test                                      # typecheck + unit + fixtures 全过
npm run release:check-plans
npm run release:verify                                      # 含 typecheck PASS + zip 断言
```

- 不重打发布件（按上次约定）；如需更新 consumer 再 `npm run release:pack`。
